# TUI UX Blueprint (Freeze Before Start)

Статус: согласовано, реализацию не начинать до явной команды "старт".

## 1) Контекст

Цель: превратить текущий CLI-вывод команд release/deploy/build в современный, информативный и управляемый TUI-опыт без потери деталей.

Режим работы: сначала UX-концепция и контракт интерфейса на моках и архитектуре, потом реализация.

## 2) Зафиксированные решения

### 2.1 Режимы входа

- Direct mode: команда запускается сразу, без preflight-экрана и без блока рисков.
- Launcher mode: отдельный экран выбора команд с обзором flow и деталями выбранной команды.

### 2.2 Launcher

- Список команд расположен слева.
- Сортировка: по основному flow.
- Формат строки команды: название + короткое описание + N steps.
- Навигация стрелками: справа обновляются детали выделенной команды.
- Enter: немедленный запуск выбранной команды.
- Порядок секций в деталях команды:
  1.  Что делает
  2.  Шаги
  3.  Пример запуска
  4.  Ожидаемый результат

### 2.3 Runtime

- Фиксированный хедер:
  - Команда
  - Текущий шаг
  - Статус
  - Горячие клавиши
- Левая колонка: только индикатор прогресса.
- Шаги слева: статус + длительность шага.
- Правая зона: лента цельных блоков шага (дашборд-элементы), не фрагменты текста.
- Блоки не сворачиваются автоматически и остаются в полном виде.
- Скролл: только у правой ленты.
- Автоскролл правой ленты: только если пользователь внизу.
- Если пользователь смотрит историю: позиция не прыгает.
- Jump to live отдельной кнопкой не нужен (используется End).

### 2.4 Log Dock

- Расположение: нижняя панель на всю ширину.
- Свернутый вид: 3 строки.
- Содержимое: все события по порядку.
- Клавиши:
  - основная: ~
  - fallback: l
- Expanded: почти на всю высоту до хедера.
- В expanded режиме: левая колонка прогресса скрывается.

### 2.5 Ошибки

- Стратегия: fail-fast.
- Требование: максимально конкретная причина ошибки.
- При ошибке: лог-док не раскрывается автоматически.
- Retry в v1: не делаем.

## 3) Концепция представления данных

Вместо "стены текста" используется Event Blocks UI:

- Слева: контекст процесса (где находимся в pipeline).
- Справа: цельные информационные блоки по событиям шага.
- Снизу: последовательный лог-поток для подробностей.

Типы блоков (выбор по структуре данных):

- Pair Grid: пары ключ-значение.
- Status List: короткие статусы.
- Files Table: измененные файлы/whitelist.
- Command Trace: выполненные команды (git/yarn/etc.).
- Git Context Block: последние коммиты, счетчики.
- Result Block: итог шага и следующий шаг.

## 4) Современная тема (предложение)

Тема: Neon Slate Ops

Дизайн-направление:

- Тон: инженерный, чистый, контрастный, без визуального перегруза.
- Фон: глубокий slate, мягкие границы, акцент только на событиях.
- Акцент: холодный cyan для activity, зеленый для done, amber для warning, red для fail.

### 4.1 Палитра токенов

- bg.base: #0b1220
- bg.elev1: #111827
- bg.elev2: #1f2937
- border.subtle: #334155
- text.primary: #e5e7eb
- text.secondary: #94a3b8
- accent.info: #22d3ee
- accent.success: #22c55e
- accent.warning: #f59e0b
- accent.error: #ef4444
- accent.focus: #38bdf8

### 4.2 Типографика и плотность

- Моноширинный рендер терминала, но с визуальной иерархией через вес и цвет.
- Вертикальный rhythm:
  - между блоками: 1 строка
  - внутри блока: 0-1 строка
- Символы статусов:
  - running: ▸
  - done: ✔
  - warning: ▲
  - error: ✖

### 4.3 Состояния интерфейса

- Idle: нейтральный slate.
- Running: cyan-акцент и мягкая анимация курсора/спиннера.
- Success: зеленый результатный маркер в хедере и финальном блоке.
- Failed: красный маркер шага + заметная причина в Result Block + запись в Log Dock.

## 5) Карта команд (v1)

Источник: package.json scripts и release-cli команды.

Ключевые команды для Launcher flow:

1. release prepare
2. release apply
3. release rollback-prepare
4. deploy-web
5. build-apk
6. rollback
7. bump-version

### Принцип детализации

- release prepare/apply: высокий уровень + структурные блоки + лог-док.
- deploy-web/build-apk: более активный realtime из-за долгих операций.
- rollback/bump-version: компактнее, но с тем же каркасом.

## 6) Архитектурный каркас компонентов

- AppShell
- HeaderBar
- CommandListPane (launcher left)
- CommandDetailsPane (launcher right)
- ProgressTimelinePane (runtime left)
- EventBlocksFeedPane (runtime right)
- LogDock

### 6.1 Каталог компонентов (v1)

1. AppShell

- Роль: общий контейнер страниц Launcher/Runtime.
- Обязанности: сетка лейаута, переключение режимов, фиксированный хедер.
- Переиспользование: общий для всех команд.

2. HeaderBar

- Роль: всегда видимый верхний бар.
- Поля: команда, текущий шаг, общий статус, горячие клавиши.
- Переиспользование: общий для Runtime, упрощенный вид для Launcher.

3. CommandListPane

- Роль: навигация в Launcher.
- Элемент списка: название + короткое описание + N steps.
- Сортировка: по основному flow.

4. CommandDetailsPane

- Роль: детальная карточка выделенной команды в Launcher.
- Порядок секций:
  - Что делает
  - Шаги
  - Пример запуска
  - Ожидаемый результат

5. ProgressTimelinePane

- Роль: левый индикатор прогресса в Runtime.
- Формат шага: статус + длительность шага.
- Ограничение: не интерактивный.

6. EventBlocksFeedPane

- Роль: правая лента цельных блоков шага.
- Поведение:
  - блоки добавляются целиком;
  - старые блоки не сворачиваются;
  - скролл только у правой ленты;
  - автоскролл только когда пользователь внизу.

7. LogDock

- Роль: подробный последовательный поток событий.
- Состояния: collapsed (3 строки), expanded (почти до хедера).
- Клавиши: ~ и fallback l.
- В expanded: левая колонка прогресса скрывается.

### 6.2 Базовые контракты данных компонентов

1. CommandMeta

- id
- title
- shortDescription
- stepsCount
- sections (does, steps, example, expectedResult)

2. RuntimeStep

- id
- title
- status (todo/running/done/failed)
- startedAt
- finishedAt
- durationMs

3. EventBlock

- id
- stepId
- type (pair-grid/status-list/files-table/command-trace/git-context/result)
- payload
- createdAt

4. LogEvent

- id
- timestamp
- level (info/warn/error/success)
- message
- stepId?

### 6.3 Правило выноса компонентов

- Если паттерн повторяется минимум в 2 командах: вынос в общий компонент.
- Если паттерн узкий и одноразовый: локально внутри команды.
- Если есть сомнение: сначала локально, вынос после второго использования.

### Принцип расширения

- Команда за командой.
- Сначала использовать существующие блоки.
- Если блока нет и он универсален, выносить в общий компонент.
- Если блок слишком узкий/одноразовый, оставлять локально в конкретной команде.

## 6.4 Командный UX-процесс (обсуждение по одной команде)

Для каждой команды из Launcher применяем один и тот же цикл:

1. Разбор команды

- смотрим фактический сценарий выполнения;
- фиксируем обязательные данные, которые нельзя терять;
- определяем, где нужен realtime, а где достаточно snapshot-блоков.

2. Дискуссия UX

- обсуждаем макет конкретно этой команды;
- выбираем типы блоков для правой ленты;
- подтверждаем поведение лог-дока именно для этой команды.

3. Консенсус

- фиксируем итоговый формат команды в этом документе;
- утверждаем список блоков в порядке появления.

4. Компонентное решение

- проверяем, есть ли подходящий компонент;
- если есть: используем;
- если нет: создаем новый общий или локальный по правилу выноса.

5. Freeze по команде

- команда помечается как согласованная;
- только после этого разрешается реализация.

## 7) Ограничения v1

- Без автоматического retry.
- Без auto-expand log на ошибке.
- Без отдельной кнопки Jump to Live.
- Без риска-блока в direct mode.

## 8) Ready Criteria перед стартом реализации

- Подтвержден список команд v1 в Launcher.
- Подтвержден набор блоков для первой команды (release prepare).
- Подтверждено, что реализация идет только через новый TUI-каркас, без ломки существующей логики команд.

## 9) Очередь обсуждения команд (по одной)

Порядок дискуссии и freeze:

1. release prepare
2. release apply
3. release rollback-prepare
4. deploy-web
5. build-apk
6. rollback
7. bump-version

Для каждой команды фиксируем отдельный mini-contract:

- Какие данные обязательны в правой ленте.
- Какие блоки используются и в каком порядке.
- Нужно ли realtime.
- Какие детали остаются только в LogDock.
- Какие компоненты переиспользуются/добавляются.
